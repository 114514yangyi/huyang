应用层

作用:提供用户或者应用程序访问互联网的接口

应用层结构

客户/服务器结构

客户端（Client）：

  • 启动方式：根据需要启动。

  • 联系服务器方式：客户端主动与服务器建立联系，客户端先发起通信。

  • 动态IP地址：客户端可能具有动态分配的IP地址。

服务器（Server）：

  • 运行方式：作为守护进程（始终运行）。
  • 提供服务方式：向客户端提供请求的服务。
  • 固定IP地址：服务器通常具有固定的IP地址。

peer-to-peer结构

核心特点：

  • 无始终在线服务器：对等网络中没有始终在线的中央服务器，而是由各个端系统直接相互通信。
  • 端系统直接通信：对等网络中的节点（对等者）可以互相请求服务，也可以相互提供服务。
  • 自我可扩展性：新的节点加入可以带来新的服务能力，同时也带来新的服务需求，因此网络具有自我扩展的性质。
  • 节点连接多变：对等节点的连接不稳定，IP地址可能会改变。
  • 高度扩展性但管理困难：对等网络具有很高的可扩展性，但管理复杂度较高。

DNS : Domain Name Service 域名解析服务

将域名解析陈相应的IP地址

层次化的域名解析

根域名服务器（Root Name Servers）：

  • 功能：根域名服务器位于 DNS 层次结构的最顶部，负责管理顶级域的信息，当本地域名服务器无法解析域名时，会联系根域名服务器。
  • 作用：根域名服务器不直接回答 DNS 查询，而是指示本地域名服务器转向哪个顶级域名服务器处理查询。

顶级域名服务器（Top-Level Domain Servers）：

  • 功能：顶级域名服务器负责管理顶级域（TLD）信息，如.com、.org、.net、.edu等，以及各国家顶级域，如.cn、.uk、.fr等。
  • 作用：当本地域名服务器接收到从根域名服务器指示后的查询，会进一步向适当的顶级域名服务器发出查询以获取更详细的信息。

权威 DNS 服务器（Authoritative DNS Servers）：

  • 功能：权威 DNS 服务器包含有关特定域的权威信息，负责提供给查询者域名到IP地址的映射。
  • 作用：当本地域名服务器需要解析特定域的信息时，会向该域的权威 DNS 服务器发送查询请求以获取准确的映射关系。

本地域名服务器（Local Name Servers）：

  • 功能：本地域名服务器通常由住宅ISP、公司或大学维护，存储已查询过的域名映射关系，提高解析效率。
  • 作用：当用户设备发起 DNS 查询时，查询会首先发送到本地域名服务器，如果本地服务器有相应的记录，将直接返回结果；否则，会按照层级结构向更高级的域名服务器继续查询。
  1. Every server knows the root
  2. Root server knows about all top-level domains

域名解析的顺序:

DNS记录

DNS （域名系统）记录是存储在 DNS 服务器中的资源记录，用于将域名映射到对应的 IP 地址或其他信息。这些 DNS 记录称为 DNS 资源记录（DNS resource record，RR），其格式通常为（名称，值，类型，生存时间）。

  • 名称（Name）：记录的域名。
  • 值（Value）：与该域名相关联的内容，比如 IP 地址或其他记录。
  • 类型（Type）：指明值的类型，表明这条记录的含义。
  • TTL（Time to Live）：生存时间，指定该记录在其他 DNS 服务器中可以被缓存的时间长度。

常见的 DNS 资源记录类型：

  1. NS（Name Server）：指定某个域名使用哪个 DNS 服务器来进行解析。
    • 例如：(networkutopia.com, dns1.networkutopia.com, NS, 32768)
  2. MX（Mail Exchanger）：指定接收该域名邮件的邮件服务器。
  3. A（Address）：将主机名解析为 IPv4 地址。
    • 例如：(dns1.networkutopia.com, 212.212.212.1, A, 5600)
  4. CNAME（Canonical Name）：为一个主机定义一个别名, 常用于重定向网址到另一个域名。

EM : 电子邮件

组成部分:

  1. 用户代理 (User Agent)：
    • 作用：用户代理是用户使用的邮件客户端，用于创建、发送、接收和管理电子邮件。这包括常见的邮件应用程序如Outlook、Gmail、Thunderbird等。用户代理提供用户界面，允许用户编写邮件、添加附件、设定接收人等操作，同时负责显示收到的邮件内容。
  2. 邮件服务器 (Mail Server)：
    • 作用：邮件服务器是专门用于接收、存储和转发电子邮件的服务器。它扮演着邮件的中转站和存储设备的角色。邮件服务器可以分为两种类型：SMTP服务器（发送邮件）和POP3/IMAP服务器（接收邮件）。SMTP服务器用于通过SMTP协议发送邮件，而POP3和IMAP服务器用于接收用户的邮件。
  3. SMTP (Simple Mail Transfer Protocol)：
    • 作用：SMTP是用于在网络上传输电子邮件的标准协议。它负责将在用户代理中创建的电子邮件从发送方传输到接收方的邮件服务器。SMTP是电子邮件系统中的“信使”，负责将邮件从一个邮件服务器转发到另一个邮件服务器，途径中可能需要经过多个中间服务器。

发送电子邮件的过程

电子邮件传递通常涉及三个阶段，从发送到接收的过程如下：

  1. 第一阶段：

    • 本地用户代理发送邮件到本地 SMTP 服务器

      ：

      • 用户代理充当 SMTP 客户端，将邮件传递给本地 SMTP 服务器。
      • 本地 SMTP 服务器充当 SMTP 服务器，接收来自用户代理的邮件并负责进一步处理该邮件。
      • 可以使用HTTP协议

  2. 第二阶段：

    • 本地服务器将邮件中继到远程 SMTP 服务器

      ：

      • 本地 SMTP 服务器变成了 SMTP 客户端，通过与远程 SMTP 服务器进行通信，将邮件中继到目标邮件服务器。
      • 在这个阶段，本地服务器起到了中继作用，将邮件传递到应该接收该邮件的远程服务器上，以便最终交付给收件人。
      • 开启TCP连接,使用remoteSMTP协议

  3. 第三阶段：

    • 远程用户代理通过邮件访问协议访问远程服务器上的邮箱

      ：

      • 远程用户代理使用诸如 POP3（邮局协议3）或 IMAP4（互联网消息访问协议4）这样的邮件访问协议，从远程服务器上的邮箱中获取已发送的邮件。
      • 这个阶段涉及接收邮件的用户在其设备上使用邮件客户端并通过邮件访问协议与远程服务器进行通信，从而检索收到的电子邮件。

  1. HTTP和网页邮件服务：

    • HTTP：除了传统的邮件访问协议外，一些 Web 邮件服务（如 Gmail、Hotmail、Yahoo! 等）也使用 HTTP 协议来提供电子邮件服务。

    • 工作方式：用户通过浏览器访问邮件服务提供商的网站，并使用基于 Web 的界面来阅读、发送和管理电子邮件。这种方式在移动设备和公共计算机上非常方便，而且不需要配置特定的邮件客户端。

接受邮件的协议

  1. POP（Post Office Protocol）：
    1. 是一种无状态的连接协议
    2. 功能：POP 是一种简单的邮件访问协议，用于授权用户与服务器之间的通信，并下载用户的电子邮件到本地设备中。
    3. 工作方式：用户代理（客户端）通过 POP 协议与邮件服务器进行交互，凭借授权信息下载邮件。邮件通常被下载到本地设备上，从服务器上移除。
  2. IMAP（Internet Mail Access Protocol）：
    1. 是一种有状态的连接协议
    2. 功能：IMAP 是更高级的邮件访问协议，相较于 POP，提供了更多功能，允许用户对服务器上存储的电子邮件进行管理和操作。
    3. 工作方式：用户代理通过 IMAP 与服务器通信，可以在服务器上管理邮件，如创建、移动、删除邮件等操作。邮件仍然存储在服务器上，用户可以通过不同设备访问并保持同步。

FTP协议: File Transfer Protocol

FTP会保持用户的连接

传输文件

  1. 建立连接：客户端与FTP服务器建立连接。这通常通过TCP协议完成，使用端口号21作为FTP控制端口。
  2. 身份验证：客户端向服务器发送用户名和密码进行身份验证。服务器验证用户提供的凭据，如果凭据有效，则允许用户访问FTP服务器上的资源。
  3. 登录成功：一旦身份验证成功，客户端和服务器之间的会话就会开始。此时，客户端可以浏览FTP服务器的文件和目录列表。
  4. 文件传输：客户端可以向服务器上传文件或从服务器下载文件。对于下载操作，客户端会发送一个请求给服务器，告诉它要下载的文件名和路径。对于上传操作，客户端会将文件发送到服务器，并在控制连接上发送相关指令告诉服务器如何处理该文件。
  5. 传输数据：服务器接收请求后，会建立一个数据连接以传输文件。数据连接是独立于控制连接的另一个TCP连接，用于实际的数据传输。在传输过程中，FTP使用不同的端口号来区分控制数据和实际数据。。

HTTP协议

URL

URL（Uniform Resource Locator）是WWW（World Wide Web）上对象的唯一标识符。它的格式通常如下：

  1. 请求的服务器IP地址
  2. 存放在服务器上面的资源路径
1

<protocol>://<host>:<port>/<path>?query_string

各个部分的含义如下：

  • Protocol（协议）：用于传输或解释对象的协议，例如http、ftp、Gopher等。这是告诉浏览器或客户端如何访问该资源的方式。
  • Host（主机）：对象所在的主机的DNS名称或IP地址。这是指明资源在哪个服务器上。
  • Port（端口）：可选的，指定服务器上的特定端口号。如果不指定端口号，则使用协议的默认端口，例如http的默认端口是80。
  • Path（路径）：包含对象的文件的路径名。这是服务器上的具体位置，指明服务器上哪里可以找到该资源。
  • Query_string（查询字符串）：发送到服务器应用程序的名称/值对。这是用于传递参数或设置给服务器的一种方式，服务器根据这些参数或设置来返回特定的资源或执行特定的操作。

使用cookies

Cookies是一种服务器发送到用户浏览器并保存在本地的数据片段。它们可以用来实现多种功能，包括：

  1. 授权（Authorization）：通过cookies，网站可以保存用户的登录状态。当用户首次登录时，服务器会创建一个包含用户ID的cookie并发送到用户的浏览器。之后的请求将携带此cookie，允许用户无需再次登录即可访问受限页面或功能。
  2. 购物车（Shopping carts）：在在线购物网站中，cookies用于跟踪用户的购物选择。当用户添加商品到购物车时，相关的产品信息会存储在cookie中。这使得用户在不同页面间跳转或在之后再次访问网站时，可以保持其购物状态。
  3. 推荐（Recommendations）：许多网站使用cookies来分析用户的浏览历史和偏好，从而为他们提供个性化的推荐内容或服务。
  4. 用户会话状态（User session state）：这是Cookies的另一个关键用途，它可以确保用户的浏览会话状态得到保持。

接收方式

  1. 非持续性连接:每一个item都要开启一次TCP(增加一个RTT)
  2. 并行请求:同时请求多个item(开启多个TCP连接)
  3. 持续性连接:只开启依次TCP连接(握手),之后传输所有的item
  4. 流水线请求:接连着请求n个item

这里是对获取 n 个小型对象的不同方法的总结:

  1. One-at-a-time:
    • 时间主要由延迟(RTT)决定
    • 需要约 2n 个 RTT
  2. m 个并发:
    • 需要约 2[n/m] 个 RTT
  3. 持久连接:
    • 需要约 (n+1) 个 RTT
  4. 流水线:
    • 需要约 2 个 RTT
  5. 流水线 + 持久连接:
    • 首次需要 2 个 RTT
    • 后续每次需要 1 个 RTT

对于获取 n 个大型对象(每个大小为 F)的情况,总结如下:

  1. One-at-a-time:
    • 时间主要由带宽(B)决定
    • 需要约 nF/B 的时间
  2. m 个并发:
    • 需要约 [n/m] F/B 的时间
    • 这种情况下假设带宽在大量用户之间共享,每个 TCP 连接获得相同的带宽
  3. 流水线和/或持久连接:
    • 需要约 nF/B 的时间
    • 这种情况下,唯一能帮助的就是获得更高的带宽

服务器端

  1. 客户端发送带有 “If-Modified-Since” 修饰符的 GET 请求
    • 客户端在请求中包含一个 “If-Modified-Since” 头,指定一个日期时间
  2. 服务器比较资源的最后修改时间
    • 服务器检查请求中的 “If-Modified-Since” 时间与资源的实际最后修改时间
  3. 服务器返回响应
    • 如果资源自上次请求后未修改过,服务器返回 “304 Not Modified” 状态码
    • 如果资源已经修改过,服务器返回 “200 OK” 状态码,连同最新的资源内

代理

  1. 客户端缓存(浏览器缓存):
    • 客户端(如浏览器)保存之前访问过的资源副本,以便下次直接从本地缓存中获取,减少网络请求。
  2. 前向代理(Forward Proxy):
    • 前向代理位于客户端所在的ISP中,代表客户端向服务器发起请求。
    • 前向代理可以缓存常用资源,减少网络流量和延迟。
    • 前向代理还可以执行内容过滤、安全检查等功能。
  3. 反向代理(Reverse Proxy):
    • 反向代理位于服务器旁,代表服务器向客户端提供内容。
    • 反向代理可以缓存常用资源,并提供负载均衡、SSL/TLS终止等功能。
    • 反向代理通常部署在服务提供商的基础设施中,隐藏了真实的服务器。
  4. 内容分发网络(CDN):
    • CDN是一个分布式的反向代理网络,将内容缓存在全球各地的服务器上。
    • 当用户请求内容时,CDN会自动将请求路由到最近的缓存服务器,减少延迟和带宽消耗。
    • CDN适用于分发大量静态内容,如图片、视频、CSS/JS文件等。

CDN: Content Distribution Network

  1. 内容托管和复制:
    • 内容所有者(如网站)将其静态资源(如图片、视频、CSS、JS等)上传到CDN提供商的服务器。
    • CDN提供商会将这些内容复制到分布在全球各地的多个边缘节点服务器上。
  2. 请求路由:
    • 当用户请求某个CDN托管的资源时,CDN的DNS服务器会根据用户的地理位置、网络状况等因素,将请求定向到最近且负载较低的边缘节点服务器。
    • 这个路由过程是自动化的,用户无需感知。
  3. 内容分发:
    • 用户的请求被路由到最近的边缘节点后,该节点服务器会迅速响应并提供所请求的内容。
    • 边缘节点服务器上都缓存有该内容的副本,因此可以快速响应用户请求,减少延迟。
  4. 缓存刷新:
    • 当内容所有者更新了资源,CDN会检测到变化并自动将最新版本的内容同步到各个边缘节点。
    • 这样可以确保用户始终获取到最新的内容。
