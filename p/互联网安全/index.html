秘钥

对称秘钥

接收方和发送者使用同一份秘钥进行加密和解密

加密算法:

  1. 凯撒密码

  2. 循环的加密模式

  3. DES加密算法:

    使用56位对称密钥，64位明文输入

    有3DES:使用3个不同的秘钥进行加密

  4. AES:

    使用128/192/256bits的秘钥加密128bits的数据块

非对称秘钥

所有人都知道接受者的公钥,只有接受者有私钥;发送者使用公钥加密,接受者使用私钥解密;

RSA算法:

  1. 密钥生成：
    • 选择两个大素数 p 和 q。
    • 计算 n = p * q，n 称为模数。
    • 计算欧拉函数 φ(n) = (p-1) * (q-1)。
    • 选择一个指数 e，使得 e 与 φ(n) 互质，通常选择一个小素数，常见的选择是65537。
    • 计算私钥指数 d，满足 (d * e) mod φ(n) = 1。
  2. 加密：
    • 将明文 M 转换为整数 m，要求 0 <= m < n。
    • 加密计算：C = M^e mod n，得到密文 C。
  3. 解密：
    • 解密计算：M = C^d mod n，得到原始明文 M。

私钥:$K^{-}{B}=(n,d)$;公钥:$K^{+}{B}=(n,e)$.

一般使用RSA算法来传递会话的对称秘钥,因为RSA虽然安全但是开销大

认证

不重数:在每一次的会话中都不会重复出现的数字,只会使用一次

数字签名:使用私钥对证书进行加密签名

哈希函数:将一长段的长度为M的信息映射到一个较短的长度为m的数字

  • MD5
  • SHA-1

数字签名

生成过程:

  1. 发送者使用众所周知的哈希函数来生成证书的信息摘要,并用私钥对其进行签名,附加上证书的明文信息,发送给接受者
  2. 接受者使用公钥将加密的摘要进行解密,再使用哈希函数对证书信息进行哈希处理,对照解密出来的摘要和生成的摘要是否相同.

颁布公钥的组织:CA

在数字证书体系中的一些关键概念包括：

  • 证书颁发机构 (Certification Authority, CA)：证书颁发机构是负责将公钥绑定到特定实体（Entity，E）的机构。
  • 实体：实体可以是个人、网站、路由器等，通过向CA注册其公钥，并向CA提供身份证明来确保其身份。
  • 注册流程：
    • 实体通过身份验证程序向CA注册其公钥。
    • CA生成一个证书，将实体的身份与其公钥绑定在一起。
    • 证书包含了实体的公钥，并由CA使用CA的私钥进行数字签名，证明“这是实体E的公钥”。
  • 数字签名：
    • CA数字签名证书，确认证书的真实性和可信度。
    • 数字签名可以确保证书在传输过程中不被篡改，并且可以被其他方验证。

因此当一个A想要获取B的公钥的时候,B就会发送给A,B的证书,A使用CA的公钥来解密证书,里面包含着B的公钥和B的信息.由于CA的公钥一定是对的,并且由CA颁发的证书也记录的一定是B的公钥.

安全邮件

可靠性(机密性)

可靠性的实现应该使用会话秘钥来加密数据,并且使用对方的公钥来加密会话私钥

完整性和认证

完整性应该对数据形成一个哈希函数摘要,使用自己的私钥签名,来证明信息是自己发送的,并且和原信息一起发送,可以验证完整性

可靠性+完整性+认证

安全TCP连接:传输层安全

使用TLS协议->https

MAC:message authentication code:消息认证码,用于消息的认证和完整性

使用一个Master Key来生成四种不同的秘钥,使用秘钥生成函数KDF(key derivation function)

  • 使用四种密钥：
    • Kc：用于客户端到服务器端发送数据的加密密钥。
    • Mc：用于客户端到服务器端发送数据的MAC密钥。
    • Ks：用于服务器端到客户端发送数据的加密密钥。
    • Ms：用于服务器端到客户端发送数据的MAC密钥。
  • 密钥通过密钥派生函数（Key Derivation Function, KDF）派生：
    • KDF接受主密钥（master secret）和可能的一些额外随机数据，用于生成新的密钥。

过程:

  1. 客户端发送协议版本号、加密算法列表和不重数A到服务器端。
  2. 服务端返回自己的协议版本号、加密算法选择、证书和不重数B到客户端。
  3. 客户端验证服务器端的证书，并生成一个前主秘钥。客户端使用服务端的公钥（从证书中获取）加密前主秘钥，并发送给服务器端。
  4. 服务端收到客户端加密的前主秘钥后，使用自己的私钥解密，获取前主秘钥。
  5. 客户端和服务端根据不重数和前主秘钥计算出主密钥，并通过KDF导出四个秘钥并开始加密通信
  6. 客户端发送所有的握手报文的HMAC
  7. 服务端发送所有握手报文的HMAC

HMAC:

HMAC（Keyed-Hash Message Authentication Code）是一种在计算机安全领域中使用的消息认证码（MAC）算法。HMAC通过在哈希函数的基础上加入一个密钥，结合消息本身进行计算，可以验证消息的完整性和真实性，防止数据被篡改。以下是HMAC的基本原理和特点：

  1. 基本原理：HMAC算法使用一个密钥和一个消息作为输入，通过哈希函数对它们进行公式计算得到一个消息认证码（MAC）。发送方将这个MAC与消息一起发送给接收方，接收方再使用相同的密钥和消息，通过哈希函数计算MAC并与接收到的MAC比对，从而验证消息的完整性和真实性。

  2. s:对称秘钥;m:信息;

    MAC=H(s+m).

  3. 特点：

    • HMAC算法基于哈希函数，常用的哈希函数包括MD5、SHA-1、SHA-256等。
    • HMAC使用了密钥进行计算，对消息进行哈希运算后再进行加密处理，保证了消息的机密性。
    • HMAC可以防止常见的攻击（如中间人攻击、伪造消息攻击等），提供了更高的安全性保障

IPsec:网络层安全

两种协议:

一、AH协议（Authentication Header）：

AH协议是一种用于提供IP通信认证和完整性保护的协议。其主要功能包括：

  1. 认证：AH协议提供了对IP数据包的来源进行认证的功能，确保接收到的数据包确实来自合法的发送方。
  2. 数据完整性保护：AH协议可以检测IP数据包在传输过程中是否被篡改或损坏，确保数据的完整性。

AH协议通过在IP数据包中添加一个认证头来实现上述功能，该头包含加密的哈希值和用于验证发送方的数字签名等信息。接收方可以通过验证哈希值和数字签名来确认数据包的来源和完整性。

二、ESP协议（Encapsulating Security Payload）：

ESP协议是一种提供更全面的IP通信安全保护的协议，它提供了加密、认证和完整性保护等功能。其主要特点包括：

  1. 加密：ESP协议可以对IP数据包的内容进行加密，保护数据在传输过程中的隐私。
  2. 认证：ESP协议也提供了对IP数据包的来源进行认证的功能，与AH协议类似。
  3. 数据完整性保护：ESP协议可以检测IP数据包在传输过程中是否被篡改或损坏。

ESP协议通过在IP数据包中添加一个加密的有效载荷来实现上述功能，该有效载荷包含加密的数据和认证信息。发送方使用加密密钥对有效载荷进行加密，接收方使用相同的密钥进行解密，并验证数字签名和哈希值以确保数据的完整性和来源。

SA : Security Association 安全关联

SA安全关联是一种在通信双方之间建立的规则集合，用于定义网络通信的安全属性。这些规则包括加密算法、认证方式、密钥管理等方面的约定，以确保通信双方能够安全地交换数据。

在发送数据之前，发送实体和接收实体之间会建立一个安全关联（SA）。这个SA是用于定义两个实体之间通信的安全属性和参数的。简单地说，它描述了如何安全地发送和接收数据。这种关联是单工的，即从发送实体到接收实体只有一个方向传输,所以一般会同时建立两条SA。这意味着，在发送实体想要向接收实体发送数据时，需要确保双方都同意并遵循相同的SA来确保数据的机密性和完整性。

分组模式

传输过程

  1. R1阶段:
    • 附加ESP尾迹到原始数据报: ESP会对原始的IP数据报进行封装。这意味着原始的IP数据报（包括其头部字段）会被保留并附加一个ESP尾迹。
    • 使用SA指定的算法和密钥进行加密: 安全关联（SA）定义了一系列参数，包括加密和认证算法以及相应的密钥。在这个阶段，使用这些参数对添加了ESP尾迹的数据报进行加密。
    • 在加密数据前附加ESP头部: 在加密后的数据前，会添加一个ESP头部。这个头部包含了关于加密数据的元信息。
  2. 创建认证MAC:
    • 使用SA中指定的算法和密钥，对加密后的数据进行认证，创建一个消息认证码（MAC）。MAC的目的是确保数据的完整性和真实性，确保数据在传输过程中没有被篡改。
    • 将MAC形成载荷附加到数据: MAC被添加到数据报中，形成一个新的载荷部分。这样接收方可以通过MAC验证数据的完整性和真实性。
  3. 创建新的IP头部和地址:
    • 由于这是一个封装过程（例如，用于VPN或隧道通信），所以需要创建一个新的IP头部和相关的字段。这些字段通常包括新的源和目标地址（这里是隧道的端点）。这个新的IP头部和数据一起发送，形成一个新的数据报，该数据报会在网络中传输

IP Sequence

IPsec（Internet Protocol Security）中的序列号（sequence numbers）是用于确保IP通信安全的重要机制。以下是关于IPsec序列号的一些详细解释：

对于新的安全关联（SA，Security Association）

当建立一个新的安全关联时，发送方将序列号初始化为0。

SPI

SPI全称为Security Parameter Index（安全参数索引），在网络通信中是一个用于标识和区分不同的安全服务或安全协议的标识符。在IPsec（Internet Protocol Security）中，SPI通常用于标识IPsec中的安全关联（Security Association，SA），以便在数据包中传递给接收方，并将其与正确的安全策略和密钥相关联。

SPI的作用包括以下几点：

  1. 唯一标识符：SPI的主要作用是作为用于识别不同的SA的唯一标识符。每个SA都有自己的SPI，这样在收到数据包时就能够根据SPI来匹配对应的SA。
  2. 关联安全参数：SPI与SA相关联，其中包括加密算法、认证算法、密钥信息等安全参数。接收方使用SPI确定如何处理收到的数据包，以便正确地解密和验证数据。
  3. 在数据包中传递：SPI通常被包含在数据包的头部或者IPsec封装的部分中。接收方接收到数据包时会检查SPI，以确定如何处理数据包中的安全信息。

总的来说，SPI是一个重要的标识符，用于在IPsec通信中唯一标识不同的安全关联，并帮助数据包正确地被解密和验证。通过SPI的使用，可以确保IPsec通信的安全性和完整性。

每次在SA上发送数据报时

  1. 发送方：
    • 增加序列号计数器。
    • 将新的序列号值放入序列号字段。
  2. 目的方：
    • 检查接收到的数据包是否是重复的。

目标

IPsec序列号的主要目标是防止攻击者嗅探（sniffing）并重新发送数据包（replay attack）。如果一个数据包被捕获并重新发送，由于序列号的变化，接收方可以轻松地识别出这是一个重复的数据包，从而拒绝它或采取其他适当的措施。此外，接收重复的、经过身份验证的IP数据包可能会破坏服务或导致数据不一致。

方法

目的方不跟踪所有接收到的数据包，而是使用一个窗口（window）机制来高效地检测重复的数据包。这个窗口通常是一个时间序列范围，接收方会检查每个到达的数据包的序列号是否在这个窗口内。如果不在窗口内，那么该数据包可能是重复的或是旧的。这样，系统可以有效地识别并处理重复的数据包，而不必检查每一个单独的数据包。这种方法的效率较高，因为不需要跟踪所有接收到的数据包。

数据库

IPsec是一套安全协议，用于保护IP网络上的通信数据的安全性和完整性。IPsec包含两个核心组件：Security Policy Database (SPD) 和 Security Association Database (SAD)。这两者在IPsec中发挥着不同的作用，并具有各自特定的功能和重要性。以下是关于它们的详细解释：

Security Policy Database (SPD)：安全策略数据库

  • 策略（Policy）: 对于给定的数据报，发送者需要知道是否应该使用IPsec。这意味着在发送数据之前，系统需要根据某些规则或条件决定是否应该对该数据进行加密或验证。这些规则或条件构成了安全策略。
  • 存储位置: 安全策略存储在安全策略数据库中（SPD）。
  • 内容: SPD中的策略可能包含源和目的IP地址、协议号等信息，用于确定哪些数据报需要使用IPsec加密或验证。

Security Association Database (SAD)：安全关联数据库

  • 状态持有: 端点在安全关联数据库（SAD）中持有安全关联状态。这些状态描述了与特定安全关联（SA）相关的信息。
  • 发送过程: 当发送IPsec数据报时，如路由器R1，会访问SAD以确定如何处理该数据报。它根据SAD中的信息来确定应该使用的加密或验证方法。
  • 接收过程: 当IPsec数据报到达路由器R2时，R2会检查数据报中的序列号（SPI），然后使用SPI索引SAD，并据此处理数据报。

简单总结：

  • SPD: 告诉你“要做什么”（即哪些数据报需要使用IPsec，使用哪种加密或验证方法）。
  • SAD: 告诉你“如何做”（即具体如何加密或验证数据报，包括使用的密钥、算法等信息）。

在实际应用中，SPD和SAD协同工作，确保IP网络上的通信数据安全、可靠。通过对这两者的管理和维护，可以实现高效的网络安全保障。

IKE

IKE（Internet Key Exchange）通常指的是Internet Key Exchange协议或者IKEv系列协议（IKEv协议的主要版本是IKEv1和IKEv2），是一种在网络通信中用于密钥交换的协议。IKE主要用于IPSec通信，为安全通信建立和维护虚拟私有网络（VPN）的安全关联。其原理包括以下几个步骤：

  1. 建立初始联系和标识符交换：IKE启动时，两端的IKE对等体通过交换标识符（如IP地址或其他标识符）来识别对方。在这个阶段，还可能会进行某些安全策略的检查。
  2. 身份验证和协商阶段：在这个阶段，IKE对等体会进行身份验证和协商。IKE协议支持多种身份验证方法，如基于预共享密钥的验证、基于证书的验证等。此外，还会协商密钥生成算法、加密算法等参数。
  3. 创建安全的网络连接和建立加密参数：完成验证后，IKE会创建一个安全的网络连接并使用安全的认证方式进行数据的加密传输。它通过基于一系列参数建立一个共享的密钥集合来实现。

综上所述，IKE的基本原理是通过一系列协议步骤来实现安全的密钥交换和虚拟网络的安全关联建立。它通过特定的身份验证机制和安全参数协商来确保网络通信的安全性和可靠性。

钥交换和虚拟网络的安全关联建立。它通过特定的身份验证机制和安全参数协商来确保网络通信的安全性和可靠性。
