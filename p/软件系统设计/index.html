软件系统设计

定义：

软件模式是将模式的一般概念应用于软件开发领域,即软件开发的总体指导思路或参照样板。

设计模式(Design Pattern)是一套被反复使用、多数人知晓的、经过分类编目的、代码设计经验的总结, 使用设计模式是为了可重用代码、让代码更容易被他人理解、保证代码可靠性。

设计模式的基本要素：模式名称，问题，解决方案，效果

设计模式一般有如下几个基本要素:模式名称、问题、目的、解决方案、效果、实例代码和相关设计模式

设计模式的分类

根据其目的(模式是用来做什么的)可分为创建型(Creational),结构型(Structural)和行为型(Behavioral)三种:

  • • 创建型模式主要用于创建对象。
  • • 结构型模式主要用于处理类或对象的组合。
  • • 行为型模式主要用于描述对类或对象怎样交互和怎样分配职责。

根据范围,即模式主要是用于处理类之间关系还是处理对象之间的关系,可分为类模式和对象模式两种:

  • • 类模式处理类和子类之间的关系,这些关系通过继承建立, 在编译时刻就被确定下来,是属于静态的。
  • • 对象模式处理对象间的关系,这些关系在运行时刻变化,更具动态性。

目标:开闭原则•指导:最小知识原则•基础:单一职责原则、可变性封装原则•实现:依赖倒转原则、合成复用原则、里氏代换原则、接口隔离原则

优缺点：

  • 创建型模式-》客户端不用知道具体的构造过程，无需关心具体创建的细节，隔离了类的生成
  • 是否违反开闭原则（灵活性，扩展性）
  • 高内聚低耦合
  • 隔离了什么和什么（解耦）
  • 什么场景不适用，使用范围有局限性
  • 在扩展时是否需要增加新的类，产生的是更多的小的对象
  • 是否需要提供额外的函数
  • 可重用性，透明性，可复用性
  • 使用的类的数量
  • 系统的复杂度和抽象性
  • 子类是否可以动态指定
  • 分层结构
  • 对用户隐藏实现细节
  • 分离抽象接口及其实现部分。
  • 虚拟代理通过使用一个小对象来代表一个大对象,可以减少系统资源的消耗,对系统进行优化并提高运行速度
  • 保护使用权限

面向对象的设计原则

原则

  1. 单一职责原则

    一个对象应该只包含单一的职责,并且该职责被完整地封装在一个类中。

    就一个类而言,应该仅有一个引起它变化的原因

  2. 开闭原则

    一个软件实体应当对扩展开放,对修改关闭。也就是说在设计一个模块的时候,应当使这个模块可以在不被修改的前提下被扩展,即实现在不修改源代码的情况下改变这个模块的行为。

    抽象化是开闭原则的关键。 • 开闭原则还可以通过一个更加具体的“对可变性封装原则”来描述,对可变性封装原则(Principle of Encapsulation of Variation, EVP)要求找到系统的可变因素并将其封装起来。

  3. 里氏代换原则

    如果对每一个类型为S的对象o1,都有类型为T的对象o2,使得以T定义的所有程序P在所有的对象o2都代换成o1时,程序P的行为没有变化,那么类型S是类型T的子类型。

    所有引用基类(父类)的地方必须能透明地使用其子类的对象

    里氏代换原则是实现开闭原则的重要方式之一,由于使用基类对象的地方都可以使用子类对象,因此在程序中尽量使用基类类型来对对象进行定义,而在运行时再确定其子类类型,用子类对象来替换父类对象。

  4. 依赖倒转原则

    高层模块不应该依赖低层模块,它们都应该依赖抽象。抽象不应该依赖于细节,细节应该依赖于抽象。

    也就是所有的东西都依赖于抽象，要针对接口（抽象）编程,不要针对实现编程

    实现开闭原则的关键是抽象化,并且从抽象化导出具体化实现,如果说开闭原则是面向对象设计的目标的话,那么依赖倒转原则就是面向对象设计的主要手段。

  5. 接口隔离原则

    客户端不应该依赖那些它不需要的接口，每一个接口应该承担一种相对独立的角色

    一旦一个接口太大,则需要将它分割成一些更细小的接口, 使用该接口的客户端仅需知道与之相关的方法即可（将大接口划分为小接口）

    每一个客户使用的接口都应该是专用的，也就是不包含哪些他不用的功能职责。接口仅仅提供客户端需要的行为

  6. 合成复用原则

    尽量使用对象组合,而不是继承来达到复用的目的。

    继承复用:实现简单,易于扩展。破坏系统的封装性;从基类继承而来的实现是静态的,不可能在运行时发生改变,没有足够的灵活性; 只能在有限的环境中使用。(“白箱”复用 )

    • 组合/聚合复用:耦合度相对较低,选择性地调用成员对象的操作;可以在运行时动态进行。 (“黑箱”复用 )

  7. 迪米特法则

    一个软件实体应当尽可能少的与其他实体发生相互作用

    迪米特法则可分为狭义法则和广义法则。在狭义的迪米特法则中,如果两个类之间不必彼此直接通信,那么这两个类就不应当发生直接的相互作用,如果其中的一个类需要调用另一个类的某一个方法的话,可以通过第三者转发这个调用。

    可以降低类之间的耦合,但是会在系统中增加大量的小方法并散落在系统的各个角落,它可以使一个系统的局部设计简化,因为每一个局部都不会和远距离的对象有直接的关联,但是也会造成系统的不同模块之间的通信效率降低,使得系统的不同模块之间不容易协调。

    每一个类都应当尽量降低其成员变量和成员函数的访问权限

    在类的设计上,只要有可能,一个类型应当设计成不变类;

    在对其他类的引用上,一个对象对其他对象的引用应当降到最低。

其他主题导入概念

可维护性较低的原因

  • 过于脆弱
  • 粘度过高
  • 复用性低
  • 过于僵硬

好的系统的特征

  • 可扩展性
  • 可插入性
  • 灵活性

重构

重构(Refactoring)是在不改变软件现有功能的基础上,通过调整程序代码改善软件的质量、性能,使其程序的设计模式和架构更趋合理,提高软件的扩展性和维护性。

原则的实战使用分析

单一职责原则

在这里里面我们需要将一个登录功能划分到不同的类中，使得能够划分为User的数据，DB的连接（这个是直接涉及到数据库而不是我们的Login），和界面的展示（也就是功能职责）。

这里直接使用聚合的方式是因为合成复用原则。

开闭原则

开闭原则就是当我们需要对系统的功能（类）进行扩展的时候，只需添加代码而不修改原有的代码；这里主要体现在我们对于多个同一类型的类写一个抽象类，然后其他的子类都继承这个父类。

里氏代换原则

在所有能够使用基类的地方都能够使用子类进行替代，也就是说我们在每一个函数接受的参数中都使用基类，然后在运行时开始使用子类进行替代

依赖倒转原则

在代码中使用抽象类,而将具体类放在配置文件中

也就是对于不同的来源和变换使用抽象类，使得底层（具体的source）和高层（main）都依赖于抽象，面对接口编程。

接口隔离原则

为不同的客户端提供宽窄不同的接口

合成复用原则

这里我们可以看到合成复用：将原来对于DB的继承替换成了聚合，因为这里的DB和具体的DAO是无关的，所以我们需要降低他们的耦合，将原来的DB使用合成复用原则，并且这里面将原来的DB替换成了抽象类，使用了开闭原则（依赖倒转原则），使得能够使用不同的DB进行初始化

迪米特法则

这里使用迪米特法则的特征是：降低一个类能够访问其他类的数量

并且假如B是A的朋友，C是B的朋友，A想要调用C，那么就需要在B中写一个小方法专门用于调用C的方法。

总结

  • 目标:开闭原则
  • 指导:最小知识原则
  • 基础:单一职责原则、可变性封装原则
  • 实现:依赖倒转原则、合成复用原则、里氏代换原则、接口隔离原

设计模式

策略模式

在一系列的模式变换中：

  1. 将有一些子类有的，一些子类没有的能力抽象为接口
  2. 同一个能力的不同表现也可以是抽象出一个抽象类
  3. 一个行为也有着状态和方法，可以作为一个类
  4. 比起继承的方式，更好的是使用合成复用原则
  5. 使用合成复用原则和开闭原则（接口隔离）的法则以后，就可以实现动态绑定（里氏代换原则）
  6. 分装变化
  7. 面向接口编程

==拥有策略抽象类==

在以下场景采用策略模式：

  1. 当多个相关类仅在行为表现上存在差异时， 策略模式可提供多种行为配置方案

  2. 需要算法不同变体时， 例如需权衡时空复杂度的算法变体， 可将这些变体实现为算法类层次结构

  3. 当算法涉及客户端不应知晓的内部数据时， 采用策略模式能避免暴露复杂的算法专用数据结构

  4. 当某类的操作中存在大量条件分支语句时， 应将相关条件分支重构为独立的策略类， 而非保留多重条件判断

工厂模式

工厂模式主要有三种形式：

  1. 简单工厂模式 (Simple Factory)

    • 也称为静态工厂方法模式
    • 类创建型模式
    • 通过一个工厂类根据传入的参数决定创建哪种产品类的实例
    • 使用if-else来选择不同的产品

  2. 工厂方法模式 (Factory Method)

    • 定义一个创建对象的接口，但让子类决定实例化哪个类
    • 工厂方法让类的实例化推迟到子类

  3. 抽象工厂模式 (Abstract Factory)

    1. 对象创建型模式
    2. 提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类
    3. 我们需要一个工厂可以提供多个产品对象,而不是单一的产品对象。
    4. 产品等级结构:产品等级结构即产品的继承结构,如一个抽象类是电视机,其子类有海尔电视机、海信电视机、TCL电视机,则抽象电视机与具体品牌的电视机之间构成了一个产品等级结构, 抽象电视机是父类,而具体品牌的电视机是其子类。
    5. 产品族:在抽象工厂模式中,产品族是指由同一个工厂生产的,位于不同产品等级结构中的一组产品,如海尔电器工厂生产的海尔电视机、海尔电冰箱,海尔电视机位于电视机产品等级结构中,海尔电冰箱位于电冰箱产品等级结构中。
    6. 当系统所提供的工厂所需生产的具体产品并不是一个简单的对象,而是多个位于不同产品等级结构中属于不同类型的具体产品时需要使用抽象工厂模式。
    7. 抽象工厂模式与工厂方法模式最大的区别在于,工厂方法模式针对的是一个产品等级结构,而抽象工厂模式则需要面对多个产品等级结构,

抽象工厂模式特别适用于需要创建一系列相关或依赖对象的场景，能够保证这些对象之间的兼容性。

建造者模式（Builder Pattern）

建造者模式的结构中还引入了一个指挥者类Director, 该类的作用主要有两个:一方面它隔离了客户与生产过程;另一方面它负责控制产品的生成过程。指挥者针对抽象建造者编程,客户端只需要知道具体建造者的类型, 即可通过指挥者类调用建造者的相关方法,返回一个完整的产品对象。

原型模式（Prototype Patern）

==自我复制==

根据其成员对象是否也克隆,原型模式可以分为两种形式:深克隆和浅克隆。

适配器模式（Adapter Patern）

类适配器：适配器类继承自目标类和适配者类

对象适配器：适配器继承自目标类，但是组合了适配者

双向适配器：适配器继承自目标类和适配者类，但是同时组合了两者

组合模式（Composite Patern）

（递归调用）

组合多个对象形成树形结构以表示“整体-部分”的结构层次

子类组合了父类

桥接模式（Bridge Patern）

抽象化(Abstraction)与实现化(Implementation)脱耦,使得二者可以独立地变化

就是当变化有多个维度的时候，抽离出来一个维度作为主要的变化维度成为一个抽象类，使用继承的方式来实现，其他的变化维度都使用组合的方式来实现。

对于存在两个独立变化维度的类可以将其分为抽象化和实现化两个角色

装饰器模式（Decorator Patern）

对客户透明的方式动态地给一个对象附加上更多的责任

和组合模式类似，但是他的子类外面裹的是一层抽象装饰器，在抽象装饰器下面还有更多的具体装饰器。

状态模式（State Patern）

==Context change State==

请求发送者与请求接收者消除彼此之间的耦合

==Invoker通过Command来给Receiver发送信息==

观察者模式（Observer PAtern）

定义对象间的一种一对多依赖关系,使得每当一个对象状态发生改变时,其相关依赖对象皆得到通知并被自动更新。观察者模式又叫做发布-订阅(Publish/Subscribe)模式、模型-视图(Model/View)模式、源-监听器(Source/Listener) 模式或从属者(Dependents)模式。观察者模式是一种对象行为型模式。

==Concrete Object变化的时候通知所有的Concrete Observer==

中介者模式（Mediator Patern）

==一个Concrete Mediator中存储多个Concrete Colleague，调节他们的交互，每一个Concrete Colleague存储他的Concrete Mediator==

模版方法模式（Template Method Patern）

==通过更改父类的一些方法来实现替换一个模版方法中一些流程的实现，父类实现一些固定的过程，由子类实现具体的可变行为==

一个设计师负责给出一个算法的轮廓和骨架,另一些设计师则负责给出这个算法的各个逻辑 步 骤 。 实 现 这 些 具 体 逻 辑 步 骤 的 方 法 称 为 基 本 方 法(Primitive Method),而将这些基本法方法汇总起来的方法称为模板方法(Template Method),模板方法模式的名字从此而来。

具体子类的基本方法将覆盖父类中定义的基本方法,子类的钩子方法也将覆盖父类的钩子方法,从而可以通过在子类中实现的钩子方法对父类方法的执行进行约束,实现子类对父类行为的反向控制。

单一职责原则

外观模式（Facade Patern）

外部与一个子系统的通信必须通过一个统一的外观对象进行,为子系统中的一组接口提供一个一致的界面

享元模式（FlyWeight Patern）

在享元模式中可以共享的相同内容称为内部状态(Intrinsic State),而那些需要外部环境来设置的不能共享的内容称为外部状态(Extrinsic State)

通常和工厂一起使用

通过工厂来产生享元对象，将每一个享元对象记录在hashmap中，享元对象中分为共享的和非共享的；有外部状态的时候就是：每一个享元对象中拥有这个外部类对象；

享元模式的核心在于享元工厂类,享元工厂类的作用在于提供一个用于存储享元对象的享元池,用户需要对象时,首先从享元池中获取,如果享元池中不存在, 则创建一个新的享元对象返回给用户,并在享元池中保存该新增对象。

代理模式（Proxy Patern）

通过引入一个新的对象(如小图片和远程代理对象)来实现对真实对象的操作或者将新的对象作为真实对象的一个替身

    1. SOA 的基本原则
      1. 服务解耦：服务之间的关系最小化，只是相互知道接口
      2. 服务契约:服务按照描述文档所定义的服务契约行事
      3. 服务封装:除了服务契约所描述内容，服务将对外部隐藏实现逻辑
      4. 服务重用:将逻辑分布在不同的服务中，以提高服务的重用性
      5. 服务组合:一组服务可以协调工作，组合起来形成定制组合业务
      6. 服务自治:服务对所封装的逻辑具有控制权
      7. 服务无状态:服务将一个活动所需保存的资讯最小化
    2. SOA 对互操作性的影响
      1. SOA 具有更高的互操作性：符合开放标准，可以更好的重用服务
      2. 支持服务的自动识别、发现、注册和调用等等
    3. SOA 对可伸缩性的影响
      1. SOA 具有更高的可伸缩性：服务自身高内聚、服务间松耦合，最小化维护的影响
      2. 但是 SOA 也会带来系统复杂度较高的问题
    4. SOA 对安全性的影响：
      1. 中间件可能会成为性能的瓶颈
      2. ESB 等中间件都可以成为被攻击的目标
      3. 多服务导致攻击的跟踪、溯源和防御成为困难。

  11. ✅ 【2019】【2023】微服务和 SOA 的区别，相同点

    1. 相同点：微服务和 SOA 都是分布式架构，微服务架构可以看作是SOA的一种实现方式，适用于更小规模和更灵活的应用系统。都包含了服务契约、服务封装、服务重用、服务组合、服务自治和服务无状态等基本特点。
    2. 微服务去掉了 SOA 架构中的 ESB(Enterprise Service Bus) 企业服务总线，采用轻量级通信机制 (HTTP、REST) 进行服务之间的通信。
    3. 微服务引入了熔断器：避免出现服务失效或网络问题等导致的级联故障。
    4. 部署和扩展：在SOA中，整个应用程序通常作为一个单元进行部署和扩展。而在微服务架构中，每个微服务可以独立部署和扩展，使得系统具有更好的弹性和可伸缩性。
    5. 服务粒度：在SOA中，服务的粒度可以比较大，一个服务可能涵盖多个相关的业务功能。而在微服务架构中，服务的粒度更小，每个服务通常专注于一个特定的业务功能。
    6. 组织和治理：SOA通常需要涉及企业范围内的中央治理和管理，**不能控制独立服务的演变。**而微服务架构强调团队的自治性，每个微服务可以由一个独立的团队进行开发和管理，更加注重服务的自治性和快速迭代。

你提供的文本描述的是 面向服务的架构（Service-Oriented Architecture, SOA） 的核心概念、组成要素、优缺点以及实现方式。以下是详细解析：

1. 什么是SOA？

SOA（面向服务的架构） 是一种软件设计范式，通过松耦合的服务（Service）组合实现业务功能。

  • 核心思想：将系统功能拆分为独立的、可复用的服务，通过标准协议（如HTTP/SOAP）交互。
  • 类比：像“点外卖”——你（消费者）通过平台（ESB）调用餐厅（服务提供者）的功能，无需关心厨房如何做菜。

2. SOA的核心组成要素

(1) 组件（Components）

角色 功能
Service Provider 提供服务（如支付服务、用户认证），通过接口（API）暴露功能。需关注性能、授权、成本等。
Service Consumer 调用服务的客户端（如电商网站调用支付服务）。
ESB（企业服务总线） 中介层，负责路由、转换消息（如将XML转JSON），解耦服务提供者与消费者。
Registry 服务注册中心（如UDDI），供服务注册和发现（类似“电话簿”）。
Orchestration Server 流程编排服务器（如BPEL引擎），协调多个服务按业务流程执行（如“下单→支付→物流”）。

(2) 连接器（Connectors）

类型 协议/技术 特点
SOAP Connector SOAP over HTTP 基于XML，强类型，适合企业级集成（如银行系统）。
REST Connector HTTP (GET/POST/PUT/DELETE) 轻量级，JSON格式，广泛用于Web API。
Asynchronous Messaging 消息队列（如RabbitMQ、Kafka） 支持异步通信，解耦生产者和消费者。

3. SOA的关键特性

✅ 松耦合：服务独立演进，通过接口交互，内部实现不影响调用方。 ✅ 复用性：同一服务可被多个系统调用（如用户认证服务复用）。 ✅ 可组合性：通过工作流语言（如BPEL）编排多个服务实现复杂业务逻辑。

4. SOA的典型应用场景

  • 企业系统集成：整合遗留系统（如ERP、CRM）到统一平台。
  • 跨部门协作：不同团队开发的服务通过ESB互联（如电商的订单、库存、物流服务）。
  • 云计算：AWS/Azure的微服务架构（如Lambda函数）本质是SOA的进化版。

5. SOA的约束与弱点（Weaknesses）

⚠️ 复杂性高：需设计服务契约、注册中心、ESB等中间件，开发成本高。 ⚠️ 性能开销：XML/SOAP协议臃肿，ESB可能成为瓶颈（对比REST更轻量）。 ⚠️ 演化失控：服务由不同团队维护，版本兼容性难保证（如V1接口废弃影响消费者）。 ⚠️ 无性能保证：服务提供者不承诺SLA（服务等级协议）时，延迟和可用性不稳定。

6. SOA vs 微服务（Microservices）

对比维度 SOA 微服务
通信协议 主要用SOAP/ESB 主要用REST/gRPC
粒度 粗粒度（如“订单服务”） 细粒度（如“支付服务中的退款子功能”）
部署 通常集中式（单体应用+ESB） 完全分布式，独立部署
技术栈 强依赖中间件（如WebLogic） 轻量级（如Spring Cloud）

7. 总结

  • SOA适合：大型企业需要整合异构系统、强调标准化和复用的场景。
  • 不适用场景：小型敏捷项目（过度设计）、高性能实时系统（中间件延迟高）。

层间关系与约束

  • 通信方向：通常严格单向调用（如展示层→业务层→数据层），避免跨层直接访问。
  • 组件归属：一个组件仅属于一个层（如“用户认证组件”必须明确归业务层或数据层）。
  • 技术隔离：不同层可使用不同技术栈（如展示层用JavaScript，业务层用Java）。

分层架构的优势

✅ 模块化：每层可独立开发、测试和部署。 ✅ 可维护性：修改一层时不影响其他层（如更换数据库只需调整数据层）。 ✅ 安全性：通过分层隔离敏感操作（如数据库访问不暴露给展示层）。 ✅ 可扩展性：可针对特定层扩展（如业务层横向扩容应对高并发）。

✅ 【2018】【2023】Layered pattern 和 Multi-tier pattern 的区别

  1. Layered Pattern 是 Module Style，而 Multi-tier Pattern 是 Allocation Style
  2. Layered Pattern 是将任务拆解成一个个处于特定抽象级别的子层次，每层为上一层提供服务，核心是关注点分离。
