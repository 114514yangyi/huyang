TCP的基础简介

套接字和端口

端口->是在报文段中用于指示OS中的具体的套接字

套接字->是运行在OS中的

套接字(Socket):

套接字是一种软件抽象,它允许应用进程与操作系统的传输层(transport layer)进行网络消息交换。套接字提供了一种标准的编程接口,使得应用程序可以独立于底层网络协议进行通信。

传输层地址: 套接字由两部分组成:<主机IP, 端口号>。这个地址组合被称为套接字。

端口:

16位数字用于区分应用程序:

  • 网络数据包的传输头部中包含源端口号和目的端口号。
  • 这些数字有助于路由器和其他网络设备识别和区分不同的应用程序。

多路复用和反多路复用

UDP协议

  1. 是无连接的协议,只根据目的IP和目的端口来将报文段分配给相应的套接字
  2. 与TCP相比，UDP是一种更轻量级的传输协议。UDP通常被用于实时应用或对延迟要求较高的情况，因为它不提供像TCP那样的顺序传送和重传机制。通过使用UDP，可以避免TCP中对包序号、重传和流控制等方面带来的开销和延迟.
  3. 是一种尽力而为的协议
  4. 有DNS,流媒体应用来使用

头信息

占用8字节

  • length:应用报文+头的长度
  • checksum:校验和

checksum:

将每16bits作为一个加数相加(包括应用报文和头),进位是将1加到个位,最终得到一个16bits的相加和以后,将取反码,得到最终的校验和

TCP协议

  1. 是有连接的,以连接为导向的协议,根据源IP地址,源端口,目的地址,目的端口来分配套接字
  2. 提供可靠的,顺序的字节流服务,流量控制

头信息

占用20字节

  • 序列号（Sequence Number）：对于sender有用,是当前发送的数据包的首位的字节序列号,我要发的数据的SQ是对方上次发送的ack序列号
  • 确认号（Acknowledgment Number）：对于receiver有用,对于接受的数据包的确认,,我要发送的ack序列号是上次接收到的序列号+报文的长度
  • HdrLen:表明头部的长度
  • 控制位（Flags）：
    • URG（紧急指针有效）：表示紧急指针字段是否有效。
    • ACK（确认序号有效）：表示确认序号字段是否有效。
    • PSH（立即传输）：表示接收方应用程序应该尽快从接收缓冲区读取数据。
    • RST（连接复位）：用于重置连接。
    • SYN（同步序号）：用于建立连接。
    • FIN（连接终止）：用于释放连接。
  • 窗口大小（Window Size）：占用16个比特，指示发送方期望接收的字节数量，用来控制流量和流量控制。
  • 校验和（Checksum）：占用16个比特，用于校验TCP头部和数据段完整性，以检测可能的错误。
  • 紧急指针（Urgent Pointer）：占用16个比特，在URG标志位设置的情况下表明紧急数据的末尾。

注意这里面的序列号都是按照Byte的顺序去划分的,即发送的数据包的开头是总共发送的数据包的第几个字节.

TCP的建立和关闭

TCP三次握手,四次挥手

简单理解TCP三次握手四次挥手（看一遍你就懂）-CSDN博客

3次握手:

  1. 第一次握手（SYN）：
    • 客户端（Client）首先向服务器（Server）发送一个特殊的TCP包，其中包含一个SYN标志（Syn请求连接）。这个包也包含客户端初始化的序列号，用来启动连接。发送该包后，客户端进入SYN_SENT状态，等待服务器的确认。
    • 发送(SYN,Sequence=ISN_A,ACK=NA)
  2. 第二次握手（SYN + ACK）：
    • 服务器收到客户端的SYN包之后，会回复一个包，其中第二次握手中同时包含了SYN标志和ACK标志（即确认号ACK加上SYN标志）。这个包表示服务器已接收到客户端的请求，并且对客户端的序列号进行了确认。此时，服务器会生成自己的序列号。发送完这个包后，服务器进入SYN_RECV状态。
    • 发送(SYN+ACK,Sequence=ISN_B,ACK=ISN+length)
  3. 第三次握手（ACK）：
    • 最后，客户端收到服务器的确认包后，也会回复一个包作为第三次握手的确认。这个包中只有ACK标志，确认服务器的序列号。在这一步完成后，客户端和服务器之间的连接建立成功，双方都可以开始传输数据。此时，客户端和服务器都进入ESTABLISHED状态。
    • 发送(ACK,Sequence=ISN_A,ACK=ISN_B+length)

四次挥手

  1. 第一次挥手（FIN）：
    • 结束方（通常是客户端或服务器一方）首先向对方发送一个FIN（Finish）消息，表示它不再有数据要发送给对方。发送这个包后，结束方进入FIN_WAIT_1状态，等待对方的确认。
  2. 第二次挥手（ACK）：
    • 接收方收到结束方发送的FIN消息后，会发送一个确认消息（ACK）作为第二次挥手的响应。此时，接收方进入CLOSE_WAIT状态,结束方进入FIN_WAIT_2阶段.
  3. 第三次挥手（FIN）：
    • 接收方发送完最后的数据后，也会向结束方发送一个FIN消息，表示它同意关闭连接。发送这个FIN消息后，接收方进入LAST_ACK状态.
  4. 第四次挥手（ACK）：
    • 最后，结束方收到接收方发送的FIN消息后，会发送一个确认消息（ACK），作为最后的确认。一旦双方都发送了ACK，连接就会完全关闭。在此时，结束方进入TIME_WAIT状态，等待一段时间后才会进入CLOSED状态，连接正式关闭。

可能发生的错误

  1. 数据包损坏(bit errors):
    • TCP 使用校验和(checksum)机制来检测数据包在传输过程中是否发生了错误。
    • 如果发现数据包已被损坏,TCP 会自动重传该数据包。
  2. 数据包丢失(why?):
    • TCP 使用序列号和确认机制来检测和重传丢失的数据包。
    • 当发送端没有收到预期的确认,就会重传数据包。
    • 超时会自动重传
  3. 数据包延迟(why?):
    • TCP 使用重传机制和拥塞控制算法来应对网络拥塞导致的延迟问题。
    • 发送端会跟踪网络状况,并动态调整发送速率,以避免缓存溢出和网络拥塞。
  4. 数据包乱序(why?):
    • TCP 使用序列号来重新排序接收到的数据包。
    • 接收端会缓存乱序到达的数据包,并根据序列号将它们重新排序。
  5. 数据包重复(why?):
    • TCP 使用序列号和确认机制来检测和丢弃重复的数据包。
    • 如果发送端收到重复的确认,就不会重传数据包

传输方式(接收ACK)

Stop and Wait

每次发送一个请求然后等待ACK,超时重传,相当于发送的窗口为1,接受的窗口为1

流水线传输

接连着发送m个数据包,等待这m个ack.

使用滑动窗口技术

滑动窗口的工作原理如下:

  1. 发送端维护一个发送窗口,表示可以发送但未被确认的数据包集合。

  2. 接收端维护一个接收窗口,表示可以接收的数据包范围。

  3. 当接收端确认窗口内的数据包时,发送端就可以将这些数据包从发送窗口中移除,并滑动窗口。

    只有当最左边的数据包被ack以后才能够移动窗口

  4. 通过动态调整窗口大小,TCP 可以实现流量控制和拥塞控制,提高传输效率和可靠性。

GBN协议与SR协议

GBN协议复习_确认分组和确认号-CSDN博客

GBN

对于发送端的重传方式是GBN:

相当于发送方的滑动窗口为n,接收方的滑动窗口为1.

为整个的发送的窗口建立一个时钟,当最左边的数据包接收到ACK以后,就重置窗口时钟.当超时的时候就将所有窗口中已经发送过的数据包全部重传.

对于接受端是Accumulate ACK

接收端的滑动窗口只有一个,就是只能够接收一个特定的数据包,当返回ACK(n)的时候,说明该数据帧和之前的所有数据帧都是已经接收.此时发送端应该要发送Sq(n+1).

SR

发送方的发送方式是SR

相当于发送方的滑动窗口为n,接收方的滑动窗口为n

对于每一个发送的数据包设置一个时钟,当发生超时的时候只重传这单个的数据包;当接收到NACK的时候,也只是重传这个数据包.

接收方的接受方式是Selective ACK

接收方的滑动窗口是n.

对接收到的每一个数据包都发送一个ACK,假如在接收方对于未正确接受的帧发送NACK

TCP的快速重传机制_快重传包括第一个-CSDN博客

因为想要唤醒重传的机制,需要等待数据包在sender端超时,所以可以使用冗余的3个ack在receiver端提醒sender端进行快速重传.

条件:3个冗余的ack->即顺序地接受到4个ack(一个ack是正常的)

情况:

  1. 在receiver端,维护一个长度为n的滑动窗口,使用类GBN的协议.(也就是在接收到其他的非ACK但是在滑动窗口中的数据包的时候,会将这个数据包进行缓存)
  2. 发送端还是继续发送滑动窗口中的数据

接收端:

每一次接受者返回ACK的时候都会发送一个Advertised Window的值(RWND),这个值应该是receiver的buffer(Window)的边界距离最后一个接受的数据比特的长度

发送端

拥塞控制:(发送端要考虑的事情)

  1. 决定怎么去分配在链路上的不同的终端的流量
  2. 动态分配流量

#根据丢包的数量来决定网络的拥塞状况

→CWND

  1. slow start :

    每一个RTT都将CWND进行double,相当于队每一个ACK都加上1; 直到CWND超过了”slow start threshold”→ssthresh

  2. “Additive Increase Multiplicative Decrease” (AIMD):

    对于每一个RTT都将CWND加一,对于每一个ACK都将CWDN加$\frac{1}{CWND}$​;

    丢包:

    • 三次冗余的ACK->$$ssthresh=\frac{CWND}{2};CWND=ssthresh$$
    • 超时->$ssthresh =\frac{CWND}{2};CWND=1$;initial slow start ;

能够很好地解决效率和公平性的问题

快速恢复

一般和快速重传一起使用

  1. 当发送端收到第三个重复确认的报文时，会更新ssthresh的值为$\frac{CWND}{2}$，然后立即重传丢失的报文段，并且设置：cwnd = ssthresh+3*SMSS，进入拥塞避免阶段。
  2. 拥塞避免阶段:当收到一个重复确认的报文时，设置cwnd = cwnd +SMSS。此时发送端可以发送新的TCP报文（如果新的cwnd允许）(发送接下来上次发送的序列号)
  3. 当收到新数据的确认时，设置cwnd=ssthresh。进入拥塞避免阶段,此时的$ssthresh=cwnd=\frac{(旧)cwnd}{2}$;

可靠数据传输协议 rdt

完全可信的信道

信道:没有差错,速率恒定,按照顺序接受

使用rdt1.0传输,即

  1. 发送方:经过调用就直接传输数据
  2. 接收方:直接接受数据,并且返回给上层

具有比特差错的信道

信道:有比特差错,按照顺序接受

使用rdt2.0,接收方具有比特差错检验:校验和

  1. 发送端:等待上层调用发送数据后,等待ACK或者NACK;加入收到ACK就直接继续等待上层调用,收到NACK,就重新发送并且继续等待.
  2. 接受端:接受数据包,进行比特差错检验,假如正确就直接返回给上层,并且返回ACK;否则就发送NACK,并且丢弃数据包.

具有比特差错的丢包信道

信道:具有比特差错,并且会丢包,非顺序接受

使用rdt3.0,也叫做比特交替协议

就是sender端交替发送包序列号为1,0的包;

receiver端交替接受1,0包并发送ack给sender端
